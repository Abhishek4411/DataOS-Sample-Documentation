Quick Start Guide

Deploy your first data product in under 15 minutes.

Prerequisitesâ€‹

Before starting, ensure you have:

  • Windows 10/11, macOS 11+, or Linux (Ubuntu 20.04+)
  • Administrator/sudo access
  • Internet connection
  • 8GB RAM minimum
  • 10GB free disk space

Step 1: Install DataOS CLIâ€‹

Windows (PowerShell)â€‹

Open PowerShell as Administrator and run:

# Set execution policy
Set-ExecutionPolicy Bypass -Scope Process -Force

# Install DataOS CLI
iwr https://get.dataos.io/install.ps1 -useb | iex

macOSâ€‹

# Using Homebrew
brew install dataos-cli

# Or using curl
curl -fsSL https://get.dataos.io/install.sh | bash

Linuxâ€‹

# Download and install
curl -fsSL https://get.dataos.io/install.sh | sudo bash

Verify Installationâ€‹

dataos-ctl version

Expected output:

DataOS CLI v2.5.0
Build: 2024.10.08

Step 2: Initialize and Authenticateâ€‹

Initialize Configurationâ€‹

dataos-ctl init

This creates the configuration directory:

  • Windows: C:\Users\<username>\.dataos\
  • macOS/Linux: ~/.dataos/

Loginâ€‹

dataos-ctl auth login

Follow the prompts to authenticate using your credentials.

Verify Authenticationâ€‹

dataos-ctl auth whoami

Step 3: Create Your First Depotâ€‹

A Depot is a connection to a data source.

Create Configuration Fileâ€‹

Create a file named my-depot.yaml:

my-depot.yaml
name: postgres-sales
version: v1
type: depot
tags:
  - sales
  - quickstart
description: "Sales database connection for quick start"
depot:
  type: postgresql
  connection:
    host: localhost
    port: 5432
    database: sales_db

Deploy the Depotâ€‹

dataos-ctl apply -f my-depot.yaml

Output:

INFO[0000] ðŸš€ applying depot:v1:postgres-sales...
INFO[0001] ðŸš€ depot:v1:postgres-sales created successfully

Verify Deploymentâ€‹

dataos-ctl get -t depot

Step 4: Create a Workflowâ€‹

Workflows orchestrate batch data processing tasks.

Create Workflow Configurationâ€‹

Create my-workflow.yaml:

my-workflow.yaml
name: daily-sales-etl
version: v1
type: workflow
tags:
  - etl
  - sales
  - quickstart
description: "Daily sales data ETL pipeline"
workflow:
  schedule:
    cron: "0 2 * * *"
    timezone: UTC
  dag:
    - name: extract-data
      spec:
        stack: flare:7.0
        compute: runnable-default
        stackSpec:
          job:
            explain: true
            logLevel: INFO
            inputs:
              - name: sales_data
                depot: postgres-sales
            steps:
              - sequence:
                  - name: extract
                    sql: |
                      SELECT 
                        order_id,
                        customer_id,
                        order_date,
                        total_amount
                      FROM sales_orders
                      WHERE order_date >= CURRENT_DATE - INTERVAL '1' DAY
            outputs:
              - name: daily_sales
                depot: dataos://icebase:sales

Deploy the Workflowâ€‹

dataos-ctl apply -f my-workflow.yaml

Check Workflow Statusâ€‹

dataos-ctl get -t workflow -n daily-sales-etl

Step 5: Create a Serviceâ€‹

Services are long-running processes that expose APIs.

Create Service Configurationâ€‹

Create my-service.yaml:

my-service.yaml
name: sales-api
version: v1
type: service
tags:
  - api
  - sales
description: "Sales data REST API"
service:
  replicas: 2
  ingress:
    enabled: true
    path: /api/sales
    stripPath: true
  stack: container:1.0
  compute: runnable-default
  stackSpec:
    image: myregistry/sales-api:v1.0
    ports:
      - name: http
        port: 8080
        protocol: TCP
    env:
      DATABASE_URL: ${SALES_DB_URL}
    resources:
      requests:
        cpu: 100m
        memory: 256Mi
      limits:
        cpu: 500m
        memory: 512Mi

Deploy the Serviceâ€‹

dataos-ctl apply -f my-service.yaml

Check Service Statusâ€‹

dataos-ctl get -t service -n sales-api

Step 6: Monitor Your Resourcesâ€‹

View Logsâ€‹

# Workflow logs
dataos-ctl log -t workflow -n daily-sales-etl

# Service logs
dataos-ctl log -t service -n sales-api -f

Check Resource Statusâ€‹

# List all resources
dataos-ctl get

# List specific type
dataos-ctl get -t workflow

# Get detailed info
dataos-ctl get -t workflow -n daily-sales-etl -o yaml

Step 7: Create a Workspaceâ€‹

Workspaces provide isolation for your resources.

Create Workspaceâ€‹

dataos-ctl workspace create -n analytics

Deploy to Workspaceâ€‹

dataos-ctl apply -f my-workflow.yaml -w analytics

List Workspace Resourcesâ€‹

dataos-ctl get -t workflow -w analytics

Common Commands Referenceâ€‹

Resource Managementâ€‹

# Apply (create/update)
dataos-ctl apply -f <file.yaml>

# Get (read)
dataos-ctl get -t <type> -n <name>

# Delete
dataos-ctl delete -t <type> -n <name>

# Logs
dataos-ctl log -t <type> -n <name>

Workspace Operationsâ€‹

# List workspaces
dataos-ctl workspace list

# Create workspace
dataos-ctl workspace create -n <name>

# Delete workspace
dataos-ctl workspace delete -n <name>

Context Managementâ€‹

# List contexts
dataos-ctl context list

# Switch context
dataos-ctl context use <context-name>

# Show current context
dataos-ctl context current

Troubleshootingâ€‹

Connection Issuesâ€‹

# Check connectivity
dataos-ctl ping

# Verify authentication
dataos-ctl auth whoami

Resource Not Startingâ€‹

# Check logs
dataos-ctl log -t <type> -n <name>

# Get detailed status
dataos-ctl get -t <type> -n <name> -o yaml

Permission Deniedâ€‹

Ensure you have the required permissions:

# Check your permissions
dataos-ctl auth permissions

Naming Conventionsâ€‹

Use descriptive, hierarchical names:

# Good
name: sales-daily-etl
name: customer-profile-api

# Avoid
name: workflow1
name: test

Tagging Strategyâ€‹

Apply consistent tags:

tags:
  - environment:production
  - team:analytics
  - domain:sales
  - criticality:high

Environment Variablesâ€‹

Use environment variables for configuration:

name: ${ENV}-sales-etl
description: "Sales ETL for ${ENV} environment"

Deploy with:

ENV=production dataos-ctl apply -f workflow.yaml
